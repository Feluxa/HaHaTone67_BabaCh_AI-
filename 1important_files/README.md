# Bank Support Sandbox

Добро пожаловать в песочницу банковской поддержки. Внутри Docker-образа живет сервис, по которому Ваш агент должен уметь ориентироваться и решать поставленные задачи.

Вы (Участники) запускаете контейнер локально, создаете `run`, исследуете HTTP API, находите доказательства по обращению клиента, выполняете только разрешенные действия и отправляете решение в evaluator (оценка решений). Сервис проверяет не стиль текста, а наблюдаемые факты: какие объекты были открыты, какие действия были сделаны, не было ли запрещенных вызовов и уложилась ли попытка в лимит.

## Быстрый старт через Docker Hub

```shell
docker pull lapamore/bank-support-sandbox:latest
docker run --rm -p 8000:8000 lapamore/bank-support-sandbox
```

Ссылка на образ: https://hub.docker.com/repository/docker/lapamore/bank-support-sandbox/general

Проверка, что сервис поднялся:

```shell
curl http://127.0.0.1:8000/health
```

Swagger UI доступен здесь:

```text
http://127.0.0.1:8000/docs
```

Публичный каталог API:

```shell
curl http://127.0.0.1:8000/meta/endpoints
```

В Swagger и каталоге есть как рабочие маршруты расследования, так и устаревшие, служебные или экспериментальные маршруты.

## Что есть в базе

- 5000+ синтетических клиентов.
- 5000+ счетов и карт.
- 50000+ транзакций и связанных ledger-записей.
- Обращения поддержки, сообщения, звонки, устройства, KYC, fraud-сигналы, лимиты, удержания, кредиты, комиссии, переводы, кешбэк и уведомления.
- Мерчанты, инциденты мерчантов, webhooks, банкоматы, ATM-операции и сервисные сбои.
- 10 заданий: 8 открытых и 2 закрытых. Закрытые кейсы открываются только заголовком `X-Case-Password`, если мы (организаторы) его выдали.

Данные синтетические и детерминированные. При новом запуске свежего контейнера состояние снова начинается с предзаполненной базы из образа.

## Как решать задачу

Один `run` соответствует одной попытке решения одного кейса.

1. Создайте run.

```shell
curl -X POST http://127.0.0.1:8000/runs \
  -H 'Content-Type: application/json' \
  -d '{"team_name":"team-alpha","metadata":{"agent":"local-runner"}}'
```

Ответ содержит `id`, например:

```json
{"id":"run_abc123..."}
```

2. Получите список кейсов.

```shell
curl http://127.0.0.1:8000/cases
```

3. Откройте выбранный кейс.

```shell
curl http://127.0.0.1:8000/cases/case_01_subscription_activation
```

Также существуют и закрытые кейсы. Если кейс закрыт и мы (организаторы) выдали пароль:

```shell
curl http://127.0.0.1:8000/cases/case_09_locked_outage_compliance_hold \
  -H 'X-Case-Password: <case-password>'
```

4. Все дальнейшие вызовы обязательно делайте с `X-Run-Id`.

```shell
curl http://127.0.0.1:8000/support/tickets/tic_7hx2kq/messages \
  -H 'X-Run-Id: run_abc123'
```


## Минимальный алгоритм для агента

1. Создать `run` через `POST /runs`.
2. Открыть `/cases` и выбрать кейс.
3. Открыть `/cases/{case_id}` и взять intake ticket.
4. Ходить по рабочим маршрутам расследования из `/meta/endpoints` и передавать `X-Run-Id`.
5. Читать обращение, профиль клиента, операции, связанные сущности и статьи базы знаний.
6. Выполнить действие, только если оно явно следует из политики и данных.
7. Отправить `POST /cases/{case_id}/evaluate` с `run_id`, `answer`, `evidence`, `actions`.
8. Забрать `/runs/{run_id}/metrics` и `/runs/{run_id}/export`.
